# Story 010.002: Migration — Ampliar Cobertura de Triggers de Auditoria

## Status
Draft

## Story

**Como** equipe de qualidade e conformidade,
**Eu quero** que todas as tabelas de negócio críticas do sistema possuam triggers de auditoria ativos,
**Para que** nenhuma alteração relevante fique sem rastreabilidade, independente do módulo utilizado.

## Contexto

Após a correção da `audit_trigger_func()` na Story 010.001, esta story amplia a cobertura de auditoria para tabelas que **não possuem triggers** definidos no `04-triggers.sql`.

### Tabelas COM triggers (já cobertos após 010.001)

| Tabela | Cobertura |
|---|---|
| `tbvelocidadenominal` | INSERT, UPDATE, DELETE |
| `tbapontamentoparada` | INSERT, UPDATE, DELETE |
| `tbapontamentoproducao` | INSERT, UPDATE, DELETE |
| `tbapontamentoqualidade` | INSERT, UPDATE, DELETE |
| `tblote` | UPDATE, DELETE (falta INSERT) |
| `tbusuario` | UPDATE, DELETE (falta INSERT) |

### Tabelas SEM triggers (alvos desta story)

| Tabela | Justificativa |
|---|---|
| `tblinhaproducao` | Cadastro crítico — mudanças em linhas impactam todo o cálculo de OEE |
| `tbfuncionario` | Dados de RH usados no PMP — admissões e demissões devem ser rastreadas |
| `tbdepartamento` | Dado mestre — alterações afetam hierarquia do sistema |
| `tboee` (snapshot) | Tabela APPEND-ONLY de OEE — somente INSERT é permitido; registrar para auditoria |
| `tbpmp_plano` | Plano mestre de produção — alterações têm impacto gerencial |
| `tbpmp_item` | Itens do PMP — alterações de quantidade/semana devem ser rastreadas |
| `tbpmp_headcount` | Headcount planejado — alterações devem ser atribuíveis |
| `tbcabecalhoturno` | Cabeçalho de turno — fechamento de turno é evento crítico |
| `tbmetaoee` | Metas OEE — mudanças de meta impactam análise de performance |

## Acceptance Criteria

1. Migration `24-ampliar-audit-triggers.sql` criada e executada sem erros no Supabase
2. Triggers de auditoria adicionados em todas as tabelas da lista "SEM triggers" acima
3. Para `tblinhaproducao`, `tbdepartamento`, `tbfuncionario`, `tbpmp_plano`, `tbpmp_item`, `tbpmp_headcount`, `tbmetaoee`: `AFTER INSERT OR UPDATE OR DELETE`
4. Para `tboee`: somente `AFTER INSERT` (tabela é APPEND-ONLY, UPDATE/DELETE bloqueados por rules)
5. Para `tbcabecalhoturno` (se existir no Supabase): `AFTER INSERT OR UPDATE OR DELETE`
6. Triggers faltantes em `tblote` e `tbusuario` complementados (adicionar cobertura de INSERT)
7. Todos os triggers nomeados como `audit_trigger` para consistência com o padrão existente
8. Comentário descritivo em cada trigger criado
9. Teste: inserir registro em `tblinhaproducao` → linha em `tbauditlog` com `operacao = 'INSERT'`
10. Teste: atualizar `tbpmp_item.qtd_planejada` → linha em `tbauditlog` com `campo_alterado = 'qtd_planejada'`, antes e depois

## Dev Notes

### Padrão de Nomenclatura

Seguir exatamente o padrão do `04-triggers.sql`:
```sql
CREATE TRIGGER audit_trigger AFTER INSERT OR UPDATE OR DELETE ON <tabela>
  FOR EACH ROW EXECUTE FUNCTION audit_trigger_func();

COMMENT ON TRIGGER audit_trigger ON <tabela>
  IS 'Auditoria ALCOA+ — <descrição do motivo>';
```

### Atenção: Tabelas com Rules APPEND-ONLY

A `tboee` já possui rules `no_update` e `no_delete`. Portanto, o trigger deve ser apenas `AFTER INSERT`. Se houver tentativa de UPDATE/DELETE, a rule bloqueia antes — o trigger nunca dispararia de qualquer forma.

### Dependência

Esta story depende da **Story 010.001** estar concluída, pois a `audit_trigger_func()` precisa estar corrigida antes de ampliar a cobertura.
