# Story 010.003: Frontend — Enviar Contexto de Auditoria nas Requisições

## Status
Draft

## Story

**Como** sistema de auditoria (ALCOA+),
**Eu quero** que cada requisição ao banco de dados inclua automaticamente o ID do usuário logado e informações do dispositivo,
**Para que** a `audit_trigger_func()` consiga atribuir corretamente **quem** realizou cada operação, cumprindo o princípio A (Atribuível) do ALCOA+.

## Contexto

A `audit_trigger_func()` usa `current_setting('app.current_user_id', true)` para identificar o usuário responsável por cada operação. Porém, atualmente o frontend **nunca envia** esse valor, resultando em `usuario_id = NULL` em todos os registros de auditoria — o que **viola diretamente o princípio ALCOA+ de Atribuibilidade**.

### Como funciona a integração com Supabase

O Supabase permite executar SQL arbitrário antes de uma transação usando `rpc()` ou `from().rpc()`. Para definir variáveis de sessão no PostgreSQL, usa-se:
```sql
SELECT set_config('app.current_user_id', '123', true);
SELECT set_config('app.device_info', '{"browser":"Chrome","os":"Windows"}', true);
```

O parâmetro `true` indica que a configuração é **local à transação** — válida apenas durante aquela requisição.

### Abordagem de implementação

Criar um **wrapper do cliente Supabase** que, antes de qualquer mutação (INSERT/UPDATE/DELETE), executa uma RPC que define as variáveis de sessão. Isso deve ser **transparente** para o restante do código — todos os serviços (`pmp.api.ts`, `supervisao.api.ts`, etc.) usam o cliente sem mudanças.

## Acceptance Criteria

1. Função RPC `set_audit_context(p_user_id INTEGER, p_device_info TEXT)` criada no PostgreSQL via migration `25-fn-set-audit-context.sql`
2. A função executa `set_config('app.current_user_id', p_user_id::text, true)` e `set_config('app.device_info', p_device_info, true)`
3. Arquivo `src/lib/supabase.ts` (ou equivalente onde o cliente é instanciado) atualizado para expor função `setAuditContext(userId: number)`
4. `setAuditContext()` chamada automaticamente **uma vez por login** (ou ao recuperar sessão) — não em cada requisição individualmente
5. Device info capturado no frontend: `{ browser, os, userAgent }` via `navigator.userAgent`
6. Após login bem-sucedido: `supabase.rpc('set_audit_context', { p_user_id: usuario.usuario_id, p_device_info: JSON.stringify(deviceInfo) })` é chamado
7. Teste: fazer login → inserir apontamento → verificar `tbauditlog.usuario_id` preenchido com o ID correto
8. Teste: verificar `tbauditlog.device_info` preenchido com info do browser
9. Nenhuma mudança necessária nos serviços existentes (`pmp.api.ts`, `supervisao.api.ts`, etc.)

## Tasks / Subtasks

- [ ] Criar migration `database/migrations/25-fn-set-audit-context.sql` (AC: 1, 2)
  ```sql
  CREATE OR REPLACE FUNCTION set_audit_context(
    p_user_id    INTEGER,
    p_device_info TEXT DEFAULT NULL
  )
  RETURNS VOID AS $$
  BEGIN
    PERFORM set_config('app.current_user_id', p_user_id::text, true);
    IF p_device_info IS NOT NULL THEN
      PERFORM set_config('app.device_info', p_device_info, true);
    END IF;
  END;
  $$ LANGUAGE plpgsql SECURITY DEFINER;

  COMMENT ON FUNCTION set_audit_context IS
    'Define contexto de auditoria para a sessão atual (ALCOA+ — Atribuível)';
  ```

- [ ] Executar migration `25-fn-set-audit-context.sql` no Supabase (AC: 1)

- [ ] Localizar onde o usuário logado é armazenado no frontend (contexto/store de autenticação) (AC: 3)
  - [ ] Identificar arquivo de autenticação (`AuthContext`, `useAuth`, ou equivalente)
  - [ ] Confirmar que `usuario_id` (INTEGER) está disponível após login

  ```typescript

    return JSON.stringify({
      userAgent: navigator.userAgent,

    ```
  - [ ] Também chamar ao **recuperar sessão existente** (ex: `onAuthStateChange` ou equivalente)

- [ ] Validar funcionamento end-to-end (AC: 7, 8)
  - [ ] Login com usuário de teste
  - [ ] Inserir ou editar qualquer apontamento
  - [ ] Consultar `tbauditlog` e confirmar `usuario_id` e `device_info` preenchidos

## Dev Notes

### Localização esperada do cliente Supabase

O cliente provavelmente está em um desses arquivos:
- `src/lib/supabase.ts`
- `src/lib/supabaseClient.ts`
- `src/services/supabase.ts`

Verificar antes de implementar.

### Por que não enviar em cada requisição?

Enviar `set_audit_context` em cada requisição geraria overhead desnecessário (2 queries por operação). Como o contexto de sessão é recuperado por cada transação do Supabase (connection pooling via pgBouncer em modo `transaction`), a abordagem mais robusta é:

1. Chamar ao login/recuperação de sessão
2. Para ambientes com pgBouncer em modo `session`, isso seria suficiente
3. Para pgBouncer em modo `transaction` (padrão Supabase), considerar chamar antes de cada mutação crítica — mas isso é otimização futura

### Alternativa mais robusta (para evolução futura)

Criar um wrapper de mutação que sempre chama `set_audit_context` antes de INSERT/UPDATE/DELETE. Mas isso está fora do escopo desta story — o objetivo aqui é a solução mínima funcional.

### Dependência

Esta story depende da **Story 010.001** (função `audit_trigger_func()` corrigida). Pode ser desenvolvida em paralelo com **Story 010.002**.
