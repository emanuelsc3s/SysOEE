# Story 010.001: Migration — Corrigir tbauditlog e audit_trigger_func()

## Status
Draft

## Story

**Como** auditor de qualidade / equipe de desenvolvimento,
**Eu quero** que a tabela `tbauditlog` e a função `audit_trigger_func()` funcionem corretamente,
**Para que** todas as operações de INSERT, UPDATE e DELETE em tabelas críticas sejam rastreadas de forma confiável e em conformidade com ALCOA+.

## Contexto e Problemas a Corrigir

A função `audit_trigger_func()` em `03-functions.sql` possui os seguintes bugs que impedem o funcionamento da trilha de auditoria:

| # | Bug | Localização | Impacto |
|---|---|---|---|
| 1 | Para INSERT/DELETE insere em `tb_audit_log` (não existe) | `03-functions.sql:212` | INSERTs e DELETEs não auditados — erro em runtime |
| 2 | `tbauditlog.registro_id UUID NOT NULL` mas tabelas novas usam `INTEGER` PK | `02-tables.sql:438` | Registro incorreto para todas as tabelas pós-refatoração |
| 3 | `tbauditlog.usuario_id BIGINT REFERENCES tbusuario(id)` — FK inválida | `02-tables.sql:451` | Todos os inserts falham por constraint quebrada |
| 4 | Captura PK como `(v_new_record).id` mas PKs são `<nome>_id` | `03-functions.sql:202` | `registro_id` nunca capturado corretamente |
| 5 | `ip_address` nunca preenchida na função | `03-functions.sql:158` | Campo existe na tabela mas sempre NULL |
| 6 | `device_info`/`user_agent` nunca preenchidos | `03-functions.sql:158` | Rastreabilidade incompleta |

## Acceptance Criteria

1. Migration `23-fix-audit-trail.sql` criada e executada sem erros no Supabase
2. Tabela `tbauditlog` recriada (DROP + CREATE) com schema correto:
   - `auditlog_id INTEGER GENERATED BY DEFAULT AS IDENTITY` como PK (padrão do projeto)
   - `registro_id TEXT NOT NULL` (suporta UUID, INTEGER e qualquer tipo de PK)
   - `usuario_id INTEGER NULL REFERENCES tbusuario(usuario_id)` (FK correta, nullable para operações de sistema)
   - `ip_address INET` capturado via `inet_client_addr()`
   - `device_info TEXT` capturado via `current_setting('app.device_info', true)`
   - Rules APPEND-ONLY recriadas: `no_update` e `no_delete`
3. Função `audit_trigger_func()` corrigida com `CREATE OR REPLACE`:
   - Para **UPDATE**: insere uma linha em `tbauditlog` por campo alterado (exceto `updated_at`, `updated_by`, `sync_data`)
   - Para **INSERT**: insere uma linha com `campo_alterado = NULL`, `valor_novo = row_to_json(NEW)::text`
   - Para **DELETE**: insere uma linha com `campo_alterado = NULL`, `valor_anterior = row_to_json(OLD)::text`
   - Ambos UPDATE e INSERT/DELETE usam **`tbauditlog`** (nome único e correto)
   - PK descoberta dinamicamente via `information_schema.table_constraints` — sem hardcode de `.id`
   - `ip_address` preenchido com `inet_client_addr()`
   - `device_info` preenchido com `current_setting('app.device_info', true)` (NULL se não configurado)
   - `usuario_id` resolvido via: `current_setting('app.current_user_id')` → fallback `updated_by`/`created_by` do registro → NULL
4. Índices recriados em `tbauditlog`:
   - `idx_audit_tabela_registro ON tbauditlog(tabela, registro_id, timestamp DESC)`
   - `idx_audit_usuario ON tbauditlog(usuario_id, timestamp DESC)`
   - `idx_audit_timestamp ON tbauditlog(timestamp DESC)`
   - `idx_audit_operacao ON tbauditlog(tabela, operacao, timestamp DESC)`
5. Teste de validação INSERT: inserir um `tbapontamentoparada` → verificar linha em `tbauditlog` com `operacao = 'INSERT'`, `valor_novo` preenchido, `registro_id` correto
6. Teste de validação UPDATE: alterar campo em `tbapontamentoparada` → verificar linha em `tbauditlog` com `operacao = 'UPDATE'`, `campo_alterado`, `valor_anterior`, `valor_novo` preenchidos
7. Teste de validação DELETE (soft): `UPDATE SET deletado = 'S'` em `tbapontamentoparada` → verificar linha em `tbauditlog` com `campo_alterado = 'deletado'`, `valor_anterior = 'N'`, `valor_novo = 'S'`
8. Triggers existentes em `04-triggers.sql` **não precisam ser recriados** — continuam funcionando pois chamam `audit_trigger_func()` que agora está correta

### DDL Completo de Referência

```sql
-- =====================================================
-- Migration 23: Corrigir tbauditlog e audit_trigger_func()
-- Data: 2026-03-01
-- Autor: Sarah (PO) / Dev
-- =====================================================

-- 1. Recriar tbauditlog com schema correto
DROP TABLE IF EXISTS tbauditlog CASCADE;

CREATE TABLE public.tbauditlog (
  auditlog_id      INTEGER GENERATED BY DEFAULT AS IDENTITY NOT NULL,
  tabela           TEXT NOT NULL,
  registro_id      TEXT NOT NULL,
  operacao         operacao_audit_enum NOT NULL,
  campo_alterado   TEXT,
  valor_anterior   TEXT,
  valor_novo       TEXT,
  usuario_id       INTEGER NULL,
  timestamp        TIMESTAMP WITH TIME ZONE NOT NULL
                     DEFAULT (now() AT TIME ZONE 'America/Fortaleza'),
  ip_address       INET,
  device_info      TEXT,
  motivo_alteracao TEXT,

  CONSTRAINT tbauditlog_pkey PRIMARY KEY (auditlog_id),
  CONSTRAINT tbauditlog_usuario_id_fkey
    FOREIGN KEY (usuario_id) REFERENCES tbusuario(usuario_id)
);

COMMENT ON TABLE tbauditlog IS 'Log de auditoria ALCOA+. APPEND-ONLY — imutável por design';
COMMENT ON COLUMN tbauditlog.registro_id IS 'PK do registro auditado convertida para TEXT (suporta UUID e INTEGER)';
COMMENT ON COLUMN tbauditlog.campo_alterado IS 'NULL = operação no registro completo (INSERT/DELETE)';
COMMENT ON COLUMN tbauditlog.device_info IS 'Info do dispositivo via app.device_info (session var)';

-- 2. Regras APPEND-ONLY
CREATE RULE audit_log_no_update AS ON UPDATE TO tbauditlog DO INSTEAD NOTHING;
CREATE RULE audit_log_no_delete AS ON DELETE TO tbauditlog DO INSTEAD NOTHING;

-- 3. Índices
CREATE INDEX idx_audit_tabela_registro ON tbauditlog(tabela, registro_id, timestamp DESC);
CREATE INDEX idx_audit_usuario        ON tbauditlog(usuario_id, timestamp DESC);
CREATE INDEX idx_audit_timestamp      ON tbauditlog(timestamp DESC);
CREATE INDEX idx_audit_operacao       ON tbauditlog(tabela, operacao, timestamp DESC);

-- 4. Corrigir audit_trigger_func()
CREATE OR REPLACE FUNCTION audit_trigger_func()
RETURNS TRIGGER AS $$
DECLARE
  v_pk_column    TEXT;
  v_record_id    TEXT;
  v_column_name  TEXT;
  v_old_value    TEXT;
  v_new_value    TEXT;
  v_usuario_id   INTEGER;
  v_ip           INET;
  v_device       TEXT;
BEGIN
  -- Capturar contexto de sessão
  v_ip     := inet_client_addr();
  v_device := NULLIF(current_setting('app.device_info', true), '');

  -- Resolver usuario_id: session var → campo do registro → NULL
  v_usuario_id := COALESCE(
    NULLIF(current_setting('app.current_user_id', true), '')::INTEGER,
    CASE WHEN TG_OP != 'DELETE' THEN (to_jsonb(NEW)->>'updated_by')::INTEGER END,
    CASE WHEN TG_OP != 'DELETE' THEN (to_jsonb(NEW)->>'created_by')::INTEGER END,
    CASE WHEN TG_OP =  'DELETE' THEN (to_jsonb(OLD)->>'updated_by')::INTEGER END
  );

  -- Descobrir coluna PK dinamicamente
  SELECT kcu.column_name INTO v_pk_column
  FROM information_schema.table_constraints tc
  JOIN information_schema.key_column_usage kcu
    ON  kcu.constraint_name = tc.constraint_name
    AND kcu.table_schema    = tc.table_schema
  WHERE tc.table_name      = TG_TABLE_NAME
    AND tc.table_schema    = TG_TABLE_SCHEMA
    AND tc.constraint_type = 'PRIMARY KEY'
  ORDER BY kcu.ordinal_position
  LIMIT 1;

  -- Obter valor da PK como TEXT
  v_record_id := to_jsonb(COALESCE(NEW, OLD))->>v_pk_column;

  IF TG_OP = 'UPDATE' THEN
    -- Registrar cada campo alterado individualmente
    FOR v_column_name IN
      SELECT column_name
      FROM information_schema.columns
      WHERE table_name   = TG_TABLE_NAME
        AND table_schema = TG_TABLE_SCHEMA
        AND column_name NOT IN ('updated_at', 'updated_by', 'sync_data')
    LOOP
      EXECUTE format('SELECT ($1).%I::text, ($2).%I::text', v_column_name, v_column_name)
        INTO v_old_value, v_new_value
        USING OLD, NEW;

      IF v_old_value IS DISTINCT FROM v_new_value THEN
        INSERT INTO tbauditlog (
          tabela, registro_id, operacao, campo_alterado,
          valor_anterior, valor_novo, usuario_id, ip_address, device_info
        ) VALUES (
          TG_TABLE_NAME, v_record_id, TG_OP::operacao_audit_enum, v_column_name,
          v_old_value, v_new_value, v_usuario_id, v_ip, v_device
        );
      END IF;
    END LOOP;

  ELSE
    -- INSERT ou DELETE: registrar operação completa
    INSERT INTO tbauditlog (
      tabela, registro_id, operacao, campo_alterado,
      valor_anterior, valor_novo, usuario_id, ip_address, device_info
    ) VALUES (
      TG_TABLE_NAME,
      v_record_id,
      TG_OP::operacao_audit_enum,
      NULL,
      CASE WHEN TG_OP = 'DELETE' THEN row_to_json(OLD)::text ELSE NULL END,
      CASE WHEN TG_OP = 'INSERT' THEN row_to_json(NEW)::text ELSE NULL END,
      v_usuario_id,
      v_ip,
      v_device
    );
  END IF;

  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

COMMENT ON FUNCTION audit_trigger_func IS
  'Captura INSERT/UPDATE/DELETE para tbauditlog (ALCOA+ compliance). Versão corrigida 2026-03-01';
```

### Observações Técnicas

- **DROP CASCADE** é necessário pois as rules e índices dependem da tabela
- Os **triggers existentes** em `04-triggers.sql` **não precisam ser recriados** — eles chamam `audit_trigger_func()` por nome, que será atualizada pelo `CREATE OR REPLACE`
- A descoberta dinâmica de PK via `information_schema` tem overhead mínimo (cache do planner). Se performance for problema futuro, pode-se usar `TG_TABLE_NAME` com um mapa hardcoded
- `SECURITY DEFINER` é necessário para que o trigger possa inserir em `tbauditlog` independente de quem disparou a operação

### Tabelas que já possuem triggers (definidas em 04-triggers.sql)

- `tbvelocidadenominal` — audit_trigger AFTER INSERT OR UPDATE OR DELETE
- `tbapontamentoparada` — audit_trigger AFTER INSERT OR UPDATE OR DELETE
- `tbapontamentoproducao` — audit_trigger AFTER INSERT OR UPDATE OR DELETE
- `tbapontamentoqualidade` — audit_trigger AFTER INSERT OR UPDATE OR DELETE
- `tblote` — audit_trigger AFTER UPDATE OR DELETE (não cobre INSERT)
- `tbusuario` — audit_trigger AFTER UPDATE OR DELETE (não cobre INSERT)
